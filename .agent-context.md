## ⚙️ Core Algorithms & Architecture

### 1. Dynamic Geocoding (Nominatim API)
- **File**: `backend/engine.py` -> `geocode_city()`
- **Mechanism**: Calls OSM Nominatim API `/search` with a custom `User-Agent`. Resolves arbitrary city strings to `[lat, lon]` and country labels.
- **Graceful Fallback**: If the geocoder fails or gets rate-limited, it automatically falls back to New Delhi (`[28.6139, 77.2090]`) as default coordinates to prevent backend errors.

### 2. Procedural Activity Generation
- **File**: `backend/engine.py` -> `generate_dynamic_activities()`
- **Mechanism**: Builds 10 themed activities matching categories (*Culture, Nature, Adventure, Food, Relaxation*).
- **Spatial Offsetting**: Offsets coordinates by random small numbers (approx -5km to +5km) using a seeded random generator (keyed on the city name) so that the coordinates are stable and map routes plot beautifully.

### 3. Constraint-Satisfaction Activity Scoring
- **File**: `backend/engine.py` -> `score_activities()`
- **Logic**:
  - Adds `+5.0` to scores if the activity category matches user preference tags.
  - Implements budget caps: **Budget** tier removes activities costing >$30; **Mid-Range** removes activities >$70.
  - Toggles weather restrictions: If weather is `rainy`, outdoor activities suffer a `-15.0` penalty, forcing indoor alternatives (e.g. museums) to the top of the recommended list.

### 4. Traveling Salesman Route Optimization (Nearest Neighbor)
- **File**: `backend/engine.py` -> `optimize_day_route()`
- **Mechanism**: For each day's planned activities, it calculates a path starting at the city center. It iteratively schedules the closest unscheduled activity using the **Haversine formula** (great-circle distance) in `haversine_distance()`, and reports cumulative travel increments.

### 5. Weather Re-routing Handshake
- **Flow**:
  1. Client clicks "Rain" button in daily timeline.
  2. JS calls `/api/replan-day` sending the target weather (`"rainy"`) and `all_other_used_ids` (ids scheduled on other days).
  3. Backend scores and filters the remaining activity pool under the rainy constraint.
  4. Returns the updated daily timeline. Client updates local state and recalculates charts and map lines.

## 🎨 UI/UX Theme Implementation Notes

- **Scenery Background**: Powered by `travel_bg.png` overlaid with `linear-gradient` dark colors.
- **Glassmorphism**: Done via CSS `backdrop-filter: blur(16px)` and semi-transparent borders.
- **Map Theme**: Instead of registering map tokens, Leaflet standard OpenStreetMap tiles are styled with an invert filter in `style.css` (`.map-wrapper` selector), transforming bright maps to premium dark-mode vectors.
- **Chart integration**: Chart.js doughnut instance is updated dynamically using `.destroy()` and re-instantiation on new trip responses.

## ☁️ Vercel Serverless Architecture

- **`vercel.json`**: Configures rewrites to route `/api/*` to `api/index.py` and everything else to the static files in `/static`.
- **`api/index.py`**: The serverless function entrypoint. It dynamically resolves the root directory and imports/exposes the ASGI `app` instance from `backend/main.py`.
- **Requirements**: Packages are resolved from the root `requirements.txt` by the Vercel builder.
